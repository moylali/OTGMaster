# VeraCrypt FAT32 Write Support Strategy

## 1. Overview
Currently, VeraCrypt encrypted USB drives are mounted as read-only. This strategy outlines the architectural and code-level changes required to implement robust, thread-safe, and native Write support for FAT32 file systems. This will be seamlessly integrated into Android's Storage Access Framework (SAF) using the `DocumentsProvider` API.

## 2. Supported Use Cases
Once implemented, the following file management capabilities will be fully supported natively inside the Android "Files" app (or any SAF-compatible file manager):
- **Creating Files and Folders**: Users can create new, empty files and directories.
- **Renaming**: Users can rename files and directories directly on the USB.
- **Deleting**: Users can safely erase files and folders from the encrypted container.
- **Sequential File Copying (Single & Multiple)**: Users can copy files from internal storage, or stream them on-the-fly from another USB drive, without intermediate caching. The process is thread-safe and can handle parallel multi-file copies gracefully.
- **Random-Access Editing (In-Place Modifications)**: By using Android 8.0's (API 26) `ProxyFileDescriptorCallback`, third-party apps (e.g., text editors, SQLite, Word, Keepass) can open files directly on the VeraCrypt drive, seek to specific byte offsets, modify them, and truncate them—all without corrupting the file or needing to cache it locally.

*(Note: Native exFAT write support will be added in a later phase, but the foundational block device layer built here will automatically support it once the filesystem driver is updated).*

## 3. Detailed Architecture Changes

### A. Cryptography Layer (C++ & JNI)
To write data to the disk, we must reverse the decryption process and encrypt the raw sectors in XTS mode before pushing them to the USB hardware.
- **`app/src/main/cpp/VeraCryptNative.cpp`**: 
  - Add a new JNI function: `Java_app_fayaz_otgmaster_veracrypt_VeraCryptNative_encryptSector`.
  - It will receive the cipher ID, master key, physical sector number (tweak), and unencrypted bytes.
  - It will perform XTS encryption using `MBEDTLS_AES_ENCRYPT` (or the equivalent adapter for Serpent/Twofish).
- **`app/src/main/java/app/fayaz/otgmaster/veracrypt/VeraCryptNative.kt`**:
  - Expose the C++ binding as `external fun encryptSector(...)`.

### B. Block Device Layer
The block device abstraction needs to support receiving writes from the `libaums` FAT32 filesystem and passing them down to the encrypted hardware.
- **`app/src/main/java/app/fayaz/otgmaster/block/RawBlockDevice.kt`**:
  - Remove the `UnsupportedOperationException` that is currently hardcoded for `writeBlocks`.
- **`app/src/main/java/app/fayaz/otgmaster/veracrypt/NativeDecryptedBlockDevice.kt`**:
  - Implement `write(deviceOffset: Long, buffer: ByteBuffer)` to intercept filesystem writes.
  - Implement `writeBlocks(startBlock: Long, data: ByteArray)`.
  - **Thread-Safety**: This method will chunk the incoming `data` into 512-byte sectors, compute the correct physical sector tweak, and invoke `VeraCryptNative.encryptSector`. It will strictly allocate a fresh, isolated `ByteArray` for every write operation to prevent data races if the Android OS attempts to copy multiple files in parallel.
  - Finally, it will push the aggregated encrypted chunks to the underlying USB hardware via `encryptedDevice.writeBlocks(...)`.

### C. Storage Access Framework Integration
The Android OS needs to know that our volume is now writable and must be able to perform standard POSIX operations.
- **`app/src/main/java/app/fayaz/otgmaster/provider/VeraCryptDocumentProvider.kt`**:
  - **`createDocument`**: Fetch the parent directory's `UsbFile` and invoke `createDirectory()` or `createFile()`.
  - **`deleteDocument`**: Fetch the target `UsbFile` and invoke `delete()`.
  - **`renameDocument`**: Fetch the target `UsbFile` and update the `name` property.
  - **`openDocument` (Write Support)**: 
    - When an external app requests `"w"` (write) or `"rw"` (read/write) mode, we will instantiate a custom `ProxyFileDescriptorCallback`.
    - This callback will intercept all native POSIX `onRead`, `onWrite`, `onFsync`, and `onRelease` events generated by the external app.
    - We will proxy these events directly to the corresponding `libaums` `UsbFile` API (`read()`, `write()`, `flush()`, and `close()`).
    - We will register the callback with Android's `StorageManager.openProxyFileDescriptor()`, which returns a fully functional, seekable `ParcelFileDescriptor` back to the app.
